ARCHITECTURAL SPECIFICATION DOCUMENTMODULE: node_supervisor (Control Plane & Intelligent Routing)I. BUSINESS OVERVIEW & INTENTnode_supervisor đóng vai trò là Hạt nhân điều phối (Control Plane) trung tâm của hệ thống Multi-Agent. Nhiệm vụ tối thượng của Node này là phân tích toàn bộ trạng thái hệ thống hiện tại—đặc biệt là các dữ liệu ngữ cảnh (Long-term Memory, User Profile) đã được nạp sẵn từ các bước trước—để đưa ra quyết định điều hướng (Routing) chính xác, phân rã bài toán và chỉ định Agent chuyên biệt kế tiếp xử lý tác vụ, hoặc ra lệnh kết thúc luồng nếu bài toán đã được giải quyết triệt để.II. COMPONENT FILES SPECIFICATION1. Core Contract (supervisor_protocol.py)Định nghĩa cấu trúc dữ liệu đầu ra bất biến của tầng nghiệp vụ điều phối, đảm bảo tính đóng băng dữ liệu khi luân chuyển qua mô hình ngôn ngữ lớn (LLM).Tiêu chuẩn kỹ thuật: Sử dụng Pydantic v2, cấu hình ConfigDict(frozen=True, extra="ignore").Cấu trúc Object: SupervisorResultnext_agent (str): Định danh chính xác Agent tiếp theo được chọn (ví dụ: "knowledge", "marketing", "smalltalk", "end").instruction (str): Chỉ thị cụ thể, chi tiết bằng ngôn ngữ tự nhiên được cá nhân hóa dựa trên profile để gửi cho Agent kế tiếp.confidence_score (float): Độ tự tin của quyết định điều hướng ($0.0 \rightarrow 1.0$).reasoning (str): Chuỗi suy luận logic giải thích vì sao Supervisor lựa chọn Agent này.2. Pure Domain Execution (supervisor_service.py)Chịu trách nhiệm cấu trúc hóa Prompt, giao tiếp với LLM cấu hình cao (High IQ Model như Claude 3.5 Sonnet / Gemini 1.5 Pro) bằng tính năng Structured Outputs để ép mô hình trả về định dạng chuẩn.Tiêu chuẩn kỹ thuật: class SupervisorService với hàm xử lý bất đồng bộ async def execute(self, history: list, user_profile: dict, relevant_episodes: list, evaluation_feedback: str, llm_engine: Any) -> SupervisorResult.Logic thực thi nội bộ:Xây dựng hệ thống Prompt tối ưu, nhúng toàn bộ Profile người dùng và các "kỷ niệm thành công" (Episodic Memory) trong quá khứ làm Few-shot examples để tối ưu hóa thuật toán suy luận của LLM.Nếu có evaluation_feedback trả về từ Node kiểm định (Evaluator) do vòng lặp trước bị lỗi, Service phải chỉ thị cho LLM phân tích feedback lỗi này để điều chỉnh hướng đi mới (Correction Loop).Gọi llm_engine.with_structured_output(SupervisorResult) để đảm bảo đầu ra khớp 1-1 với giao ước Protocol.3. Mainboard Gatekeeper (adapter_supervisor.py)Chốt chặn kết nối Graph Topology, bóc tách Bus và chuyển giao thông tin điều phối một cách an toàn.Tiêu chuẩn kỹ thuật: Hàm xử lý đồ thị async def node_supervisor(state: MainBus, config: RunnableConfig = None) -> dict.Tuân thủ Safe Post-Guard (Chống sập Đồ thị):Kiểm tra tính hợp lệ của Node đứng trước trên mạng Bus (trong luồng tuần tự từ đầu, đó là BusRegistry.INPUT_GUARD; trong luồng sửa lỗi của vòng lặp, đó là BusRegistry.EVALUATOR hoặc BusRegistry.HUMAN_REVIEW).Nếu phát hiện Node thượng nguồn không tồn tại hoặc có trạng thái "FAILED", lập tức thực hiện Hạ cánh an toàn (Graceful Degradation): Đóng gói trạng thái "FAILED" kèm theo thông tin lỗi chi tiết đẩy lên Bus qua StandardFrame.emit, hướng luồng chạy về Node xử lý lỗi chung hoặc kết thúc đồ thị an toàn thay vì dùng câu lệnh raise.Dependency Injection & Context Extraction: Gọi await get_ctx() để lấy llm_factory.get_model("default"). Sử dụng toán tử chấm (.) trích xuất an toàn user_profile và relevant_episodes từ payload của các Node lưu trữ trên Bus.BodyFrame Schema Mapping (Strict Constraint): Đóng gói dữ liệu ra Bus qua 8 trường hợp lệ duy nhất của BodyFrame:status: "SUCCESS" nếu LLM trả về quyết định hợp lệ, hoặc "FAILED" nếu gặp lỗi suy luận / lỗi API.text: Chuỗi nội dung chỉ thị của Supervisor gửi cho các Agent sau (instruction).records: Không xử lý (để trống).entities: Không xử lý (để trống).state: Lưu trạng thái runtime phục vụ Conditional Edges điều hướng: {"next_action": result.next_agent, "process_completed": False}.metrics: Lưu thông số hệ thống {"confidence_score": result.confidence_score}.context: Lưu thông tin chuỗi suy luận logic {"reasoning": result.reasoning} phục vụ truy vết (Tracing).error: Chuỗi thông báo lỗi chi tiết nếu status == "FAILED".III.     - Step 4: Status Normalization & Bus Emit - Đồng bộ hóa trạng thái 'next_action' phục vụ rẽ nhánh đồ thị và phát StandardFrame lên mạng Bus hệ thống.
    ======================================================================
    """
IV. STRICT LAWS FOR AI INTEGRATION (ĐIỀU KHOẢN CẤM)TUYỆT ĐỐI KHÔNG tự ý tạo thêm field ngoài danh sách 8 trường quy định của BodyFrame.CẤM trả về cấu trúc phân cấp phức tạp trong state hoặc context, tất cả dữ liệu điều hướng (như tên agent tiếp theo) phải nằm phẳng (Flat structure) trong state["next_action"] để các hàm Conditional Edges của LangGraph có thể đọc trực tiếp bằng code thuần.CẤM sử dụng lệnh raise phá vỡ vòng lặp đồ thị. Nếu LLM bị timeout hoặc trả về dữ liệu lỗi cấu trúc, Adapter phải bắt (catch) ngoại lệ, gán status="FAILED", ghi nhận thông tin vào trường error và dùng StandardFrame.emit để đẩy dữ liệu lên Bus.
